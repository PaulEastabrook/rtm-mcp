# Debrief — A2.1 Piece 0b shipped (server half): overlay-refresh mark

The server half is **built, tested, merged**: rtm-mcp **v1.8.0**, PR #13 → `main`
(merge commit `6677715`, feature commit `fae242c`). Merged `main` is green: **517 tests**, ruff +
pyright clean. This debrief gives the gtd side exactly what to consume.

## 1. The new durable signal (what the gtd drain consumes)

`gtd_apply_canvas_commit` now stamps **`#ai_overlay_refresh_needed`** on the **project task** after
**any successful, non-empty commit** — adds / edits / execute / notes / completes / removes. It is
the **commit-path twin** of `#ai_project_needs_finalise` (the create mark): same blackboard pattern,
server stays vault-free.

- **Where:** `rtm.tasks.addTags` on the project, inside `if applied:`, immediately after the
  `COMMIT` audit note. Best-effort + idempotent (addTags dedupes); recorded in the transaction log
  (undoable via `batch_undo`) like every other write.
- **Zero-apply commits stamp nothing** — a commit that applied no writes made no plan change, so no
  refresh is needed.
- **Distinct from the existing tags:** `#ai_progress_requested` / `#ai_progress_deferred` are the
  execute work-list; `#ai_overlay_refresh_needed` is the "the persisted overlay is stale, recompute
  it" signal. They are independent and can co-occur.

## 2. Consumer contract — what the gtd side must build (still open)

The brief's out-of-scope list, restated as the contract this server half now expects:

1. **`gtd-project-finalise` drains the mark.** For each project tagged `#ai_overlay_refresh_needed`:
   fetch the project envelope (e.g. via `gtd_project_plan` / the connector) → run
   `plan_graph_refresh.py` to **recompute + persist** the enriched AI-Memory overlay (DAG +
   judgement + timeline order + the `manual_order` pin) → then **`removeTags`** the mark. Removal is
   the "done" signal; leaving it set means "still stale".
2. **The artifact fires the finalise/refresh task on every commit** (not only execute commits), so
   the drain runs promptly rather than waiting for the next progression / weekly-review event.
3. **`tag-taxonomy.md`** documents `#ai_overlay_refresh_needed` (the brief marked this done).

Note the engine now has **two marks to drain**, both server-written, both gtd-drained:
- `#ai_project_needs_finalise` (create) → the discipline tail (vault folder, `context.md`, `_focus`,
  reverse `CONTEXT` note, progression fan-out).
- `#ai_overlay_refresh_needed` (commit) → overlay recompute + persist.
They can be drained by the same engine pass or separately; they are independent signals.

## 3. Strict-tag gate + activation (already satisfied)

`collect_commit_tags` includes `#ai_overlay_refresh_needed` whenever any actionable op is present
(so the up-front existence gate covers it — including a completes-only or removes-only commit). It
is a NEW tag, so under strict-tag mode an unprovisioned tag would reject every non-empty commit.
**Confirmed live (2026-06-27): the tag IS provisioned** in Paul's RTM account — so the MCP can be
refreshed/activated safely; no commit will be rejected by the gate.
