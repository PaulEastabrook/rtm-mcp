# Debrief — A2.1 Piece 1 shipped: `gtd_create_project`

Piece 1 (the gating server lift) is **built, tested, merged**: rtm-mcp **v1.7.0**, PR #12
(`feat/gtd-create-project` → `main`, merge commit `2b96675`). 512 tests green; ruff + pyright
clean. This debrief gives you (a) the **as-built tool contract** Piece 2 must call, (b) the
**decisions taken** to fold into the design-of-record, (c) **two deviations from the brief**, and
(d) the **remaining pieces** with their wiring contract.

## 1. As-built tool contract (what Piece 2 calls)

MCP name: **`mcp__rtm__gtd_create_project`**. Single governed, validate-then-apply write
(undoable via `batch_undo`). Read-then-write: one `rtm.tasks.getList(status:incomplete)` then the
creation writes.

```
gtd_create_project({
  frame: { life, focus, name, outcome },     # life ∈ work|personal|leanworking; name required;
                                             # focus = area NAME or area task id; outcome → INCEPTION note
  items: [ {
    id,                          # in-draft id deps reference (defaults to array index)
    type,                        # action | waiting_for | calendar  (required)
    text,                        # title
    classifiers: { context?, comms?, priority?, quick? },
    calendar_date? | chase? | due?,   # ONE date field
    start?, estimate?,
    deps: [ in-draft ids ],
    done?,                       # true → create-then-complete
    execute?,                    # now | later | quick  (durable progression signal)
    notes?: [ { title?/type?, text } ]
  } ],
  notes: [ { title?/type?, text/body } ]      # project-level notes (e.g. authored INCEPTION)
}) -> {
  project_id, url,                            # url = #list/{list}/{area}/{project}
  created: [rtm ids], completed: [rtm ids], progressed: { in-draft id: mode },
  applied: [...], errors: [...], message
}
# ambiguous focus → { candidates: [{id, name, list_id}] }   (re-call with frame.focus = one id)
# rejected        → { created: [], rejected: [...], message } (NOTHING written)
```

All three complex params accept structured JSON **or** a JSON-string (the `tool_params` Annotated +
`coerce_json` pattern), so the artifact's `callMcpTool` can pass either.

## 2. Decisions taken this session (please absorb into the design-of-record)

1. **Scope = Piece 1 only** (server tool), per Paul.
2. **Deps ARE persisted** as active `DEPENDS-ON` notes (mapping each in-draft producer id → the
   created RTM id), so the new project's canvas shows its dependency graph on first reload. Body
   format the server's reader round-trips (use this verbatim if any gtd-side code also writes deps):
   ```
   DEPENDS-ON
   Upstream RTM IDs:
     task_id: "<producer rtm id>"
     list_id: "<producer rtm list id>"
   Status: active
   ```
3. **The SERVER writes the finalise mark** `#ai_project_needs_finalise` on every create (Paul chose
   this over the brief's "or leave it gtd-side" option). **Provisioned in RTM** (confirmed
   2026-06-27), so creates pass the strict-tag gate. The progression **fan-out itself stays
   gtd-side** — the server only writes the durable tag.
4. **Every item carries `#ai_conversation`**; the project carries `life + #project + #ai_conversation
   + #ai_project_needs_finalise`. `execute` mirrors the commit tool exactly (now/quick →
   `#ai_progress_requested`, later → `#ai_progress_deferred`; a blocked item also gets
   `#ai_deferred_pending_unblock`). `later` requires `#ai_progress_deferred` (gated only when
   present — now/quick-only creates stay backward-compatible).

## 3. TWO deviations from the brief (action for your design docs)

- **Item schema is VERBOSE, not the brief's compact keys.** The brief proposed
  `{ k, t, c, m, p, deps, done, execute, ... }`. As-built uses the **same verbose shape as
  `gtd_apply_canvas_commit`'s `adds`** (`type` / `text` / `classifiers:{context,comms,priority,
  quick}` / `calendar_date|chase|due`) **+ create extras** (`id`, `start`, `estimate`, `deps`,
  `done`, `execute`, `notes`). Rationale: reuses `classifiers_to_tags` / `priority_to_code` /
  date handling verbatim — one taxonomy, one code path with commit. **Piece 2's artifact must map
  the draft → this verbose shape** (the canvas seed uses compact keys, so a small adapter is needed
  in `draftCreate()`). If you'd rather keep the compact contract, a thin server-side normaliser is
  a quick follow-up — say the word.
- **Children are created directly under their parent** (`rtm.tasks.add` with `parent_task_id`),
  **not** create-on-Processed-then-reparent. Reason: create attaches notes to the new tasks, and a
  reparent would invalidate the add-response `list_id`, breaking those note writes. Net effect:
  **no `Processed` staging list is used by create** (commit still uses it). No impact on the
  artifact; just noting it diverges from the brief's "created on Processed → reparent" line.

## 4. Area-of-Focus resolution (how `frame.focus` works)

Areas of focus carry **no marker tag** — the server derives them as the **parents of `#project`
tasks**. So `frame.focus`:
- an explicit **area task id** → used directly (works even for an area with no projects yet);
- otherwise a **name**, matched (exact → substring) among those derived areas; ambiguous → a
  candidate list; a miss → an actionable error (it **never creates loose**).

For the artifact: prefer passing the **area id** when the draft knows it; fall back to name.
